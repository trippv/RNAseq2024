Unidad IV. Ensamble de un transcriptoma de novo usando Trinity

Ahora vamos a realizar un ensamble con los reads que fueron depurados en la unidad anterior.

En este caso, los reads depurados ya fueron previamente procesados. Vamonos a curso2021. Dentro de la carpeta TRANSCRIPTOMICA vamos a crear un directorio llamado ENSAMBLE.

$ cd TRANSCRIPTOMICA
$ mkdir ENSAMBLE # creamos el directorio.
#VAMOS  CREAR UN SOFTLINK A LOS READS DEPURADOS
$ ln -s /LUSTRE/bioinformatica_data/genomica_funcional/Tripp/Curso_2020/TRIMMOMATIC/TRIM_P .

Ahora vamos a concatenar todos los reads R1 y R2 en archivos individuales y los vamos a comprimir con gzip.

$ cat TRIM_P/*1P* | gzip > R1.fastq.gz 
$ cat TRIM_P/*2P* | gzip > R2.fastq.gz

Les deben de aparecer ahora sus archivos en su actúal path.

[curso01@omica ENSAMBLE]$ ls
0.ensamble.slurm         cat_reads     R1.fastq.gz  slurm-167487.err  T1_rep1_1P.fastq  TRIM_P
1.ensamble_prueba.slurm  prueba_R1.gz  R2.fastq.gz  slurm-167487.log  T1_rep1_2P.fastq  Trinity_ensamble_resultado

Una vez que hemos creado los reads R1 Y R2, vamos a crear un script en bash para correrlo en slurm. Para correr el script debemos estar en ENSAMBLE/

$ nano 0.ensamble.slurm

#!/bin/bash
# esta línea se conoce como Shebang, y se usa para indicarle que el siguiente script
# se debe de interpretar con bash

# Directivas
#SBATCH --job-name=Trinity-denovo # nombre del trabajo que aparecera en la cola
#SBATCH --output=slurm-%j.log # salida estandar num de trabajo
#SBATCH --error=slurm-%j.err # salida estandar de error
#SBATCH --nodes 1 # número de nodos
#SBATCH --mem=20GB # RAM, 1GB por millon de reads.
#SBATCH --ntasks-per-node=8 # número de tareas por nodo
#SBATCH -t 6-00:00:00 # número maximo
#SBATCH -p cicese #cola

#my vars
PH=`pwd`
R1="$PH/R1.fastq.gz"
R2="$PH/R2.fastq.gz"
OUTDIR="Trinity_ensamble"
TRINITY="Trinity"
LIBT=RF
PARM=" --no_version_check \
--seqType fq --max_memory 20G \
--CPU 8 --left $R1 --right $R2 \
--normalize_reads \
--SS_lib_type $LIBT \
--output $OUTDIR" \
#--full_cleanup"

# Run trinity
$TRINITY  $PARM

Cuando corremos el script se van a generar dos archivos: slurm-167499.err, slurm-167499.log, el primero nos mostrará los errores que se generen, y el .log tendrá la salida estandar re-dirigida.

Una vez que termine (se va a tardar como 12 hrs en terminar, también depende de cómo se administre el calendario de tareas). El resultado lo encontramos dentro de la carpeta Trinity_ensamble, en un archivo llamado Trinity.fasta(solo si lo corren sin la opción de --full_cleanup), de otra forma aparecerá en el actúal path.

Valoración del ensamble

A continación vamos analizar algunas estadisticas que nos ayudan a saber si nuestro ensamble resultó bueno.

Acá les dejo un link https://github.com/trinityrnaseq/trinityrnaseq/wiki/Transcriptome-Assembly-Quality-Assessment de los autores de Trinity donde recomiendan una serie de cosas para analizar la calidad. En esta ocasión usaremos Transrate para evaluar nuestro ensamble dado que es el más completo.

Vamos a crear una carpeta llamada TRANSRATE dentro de la carpta ENSAMBLE:

$ mkdir TRANSRATE
$ nano 1.transrate.slurm
  #!/bin/bash

#SBATCH -p cicese
#SBATCH --job-name=transrate
#SBATCH --output=transrate-%j.log
#SBATCH --error=transrate-%j.err
#SBATCH -N 1
#SBATCH --mem=40GB
#SBATCH --ntasks-per-node=8
#SBATCH -t 06-00:00:00

TRANS=/LUSTRE/bioinformatica_data/genomica_funcional/Laura/bin/transrate-1.0.3-linux-x86_64
ENSAMBLE="../Trinity_ensamble.Trinity.fasta"
R1="../R1.fastq.gz"
R2="../R2.fastq.gz"

cd $SLURM_SUBMIT_DIR

$TRANS/transrate \
--assembly $ENSAMBLE \
--left $R1 \
--right $R2 \
--threads 8 \
--output gcontigs

Ahora corremos el script usando sbatch:

$ sbatch 1.transrate.slurm

De nueva cuenta nos va a generar dos archivos *.log y el *.err. Se va a crear una carpeta llama gcontigs donde encontraremos una seríe de archivos.
